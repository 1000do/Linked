UC-84. View Chat History

Function Trigger: Actor clicks on a conversation card in the sidebar.

Function Description: Allows the Actor to view the entire message exchange history of a selected conversation, which automatically marks unread messages in the room as read.

Function Details

  • Preconditions
    • Actor must be successfully authenticated as a Staff member or Admin.
    • Actor must have access to the selected chat room.
  • Process
    • Select Conversation:
      • Actor accesses the Chat page under the Admin Panel dashboard.
      • Actor clicks a conversation card in the sidebar chat list.
      • The system hides the empty chat state panel.
      • The system displays the chat room panel.
      • The system updates the chat header with partner name, avatar, and online status.
    • Retrieve History:
      • The system requests the backend to retrieve the chat history.
      • The server verifies that the active Actor has permission to access the chat (either as a participant or through a support/moderation override).
      • The server marks all unread messages in the chat for the active Actor as read in the database.
      • The server clears the unread count for the active Actor in the cache.
      • The server retrieves the conversation messages from the database.
      • The server filters out messages sent before or on the participant's cleared timestamp.
    • Render History:
      • The system receives the message history payload.
      • The system renders the messages in the message list viewport.
      • The system scrolls the message list viewport to the bottom.
      • The system broadcasts a read notification to other participants.
  • Alternative Flow (Render Empty Chat Room)
    • If the message history payload is empty:
      • The system receives an empty message list payload.
      • The system displays a start-chatting message in the viewport.
      • The system scrolls the message list viewport to the bottom.
  • Postconditions
    • The chat history is displayed to the Actor.
    • All unread messages in the conversation for this participant are marked as read in the database and cache.
  • Error Handling
    • Session Invalid (Not Logged In): The system displays an authentication error message and redirects the Actor to the login page.
    • Unauthorized Access: The system detects that the Actor is not a Staff or Admin, displays an access denied error page, and terminates the flow.
  • Security Measures
    • Access Permission Verification: The server verifies that the requesting Actor is an authorized participant in the conversation or has a support/moderation override before returning the message logs.
    • Message Temporal Filtering: The system filters out any messages sent before or on the participant's cleared timestamp to maintain privacy boundary integrity.
