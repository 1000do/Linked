UC-77. Send Notifications

Function Trigger: Actor clicks the "Create Notification" button, enters recipient and message details, and submits the form.

Function Description: Allows the Actor to compose and send notifications either to a specific registered user (using email tags) or broadcasted to all users.

Function Details

  • Preconditions
    • Actor must be successfully authenticated as a Staff member or Admin.
  • Process
    • Open Dialog:
      • Actor accesses the System Notifications page.
      • Actor clicks the "Create Notification" button.
      • The system opens the "Compose New Notification" modal dialog.
    • Compose Notification:
      • Actor selects "Specific User (Email)" from the Recipient Group dropdown list.
      • Actor types the recipient's email address in the tag box.
      • Actor enters the notification title (maximum 255 characters).
      • Actor enters the notification message content.
    • Send Notification:
      • Actor clicks the "Send Notification" button.
      • The system requests the backend to send the advanced notification.
      • The server validates that the active user possesses a Staff or Admin role.
      • The server validates that the recipient list contains valid registered emails and does not include the sender's own email.
      • The server validates the length constraints of the title.
      • The server saves the notification record to the database for each recipient user.
      • The server broadcasts the new notification event via SignalR.
      • The system displays a success popup message indicating the notification was successfully sent.
      • The system closes the modal dialog and refreshes the sent notifications table list.
  • Alternative Flow (Broadcast to All Users)
    • If the Actor broadcasts to all users:
      • Actor selects "Broadcast (All Users)" from the Recipient Group dropdown list.
      • The system hides the Recipient Email tag input box.
  • Alternative Flow (Autocomplete Email Suggestions)
    • If the Actor types email characters:
      • Actor types characters in the Recipient Email input box.
      • The system queries matching registered emails from the server.
      • The system displays a dropdown list of matching emails that are not already selected.
      • Actor clicks an email option from the dropdown.
      • The system adds the email as a tag pill in the tag box.
  • Alternative Flow (Add Multiple Emails)
    • If the Actor adds multiple emails:
      • Actor types an email address and presses Enter or selects it from the suggestions.
      • The system adds the email tag pill in the tag box.
      • Actor continues typing additional email addresses.
  • Alternative Flow (Remove Email Pill)
    • If the Actor removes an email:
      • Actor clicks the close (X) icon on one of the email tag pills.
      • The system removes the tag pill from the recipient list.
  • Postconditions
    • The notification record is created and stored in the database.
    • The SignalR hub pushes the notification in real time to any active online recipients.
    • The system updates the sender's sent notifications audit log table.
  • Error Handling
    • Session Invalid (Not Logged In): The system displays an authentication error message and redirects the Actor to the login page.
    • Unauthorized Access: The system detects that the Actor is not a Staff or Admin, displays an access denied error page, and terminates the flow.
    • Missing Recipient Email: The Actor attempts to submit the form under the "Specific User" group without adding any email tags, displays a warning notification indicating "Please select at least one recipient email", and returns to composing the notification.
    • Self-Notification Blocked: The Actor attempts to add their own email address as a recipient, displays a warning notification: "You cannot send a notification to yourself", and returns to composing the notification.
  • Security Measures
    • Role-based Access Control: Access to compose and send notifications is restricted strictly to authenticated users with Staff or Admin roles.
    • Self-Notification Prevention: The server validates that the recipient email does not match the sender's own email.
    • Recipient Validity Validation: The server validates that all recipient emails belong to valid registered users.
