UC-97. Reject Instructor Application

Function Trigger: Actor clicks the "Reject" button next to a pending application.

Function Description: Allows the Actor to reject an instructor's pending registration application. A detailed rejection reason is required, which is recorded in the database.

Function Details

  • Preconditions
    • Actor must be successfully authenticated as a Staff member or Admin.
    • The instructor's application must exist in a pending status.
  • Process
    • Locate Application:
      • Actor accesses the instructor registration requests page.
      • Actor locates the desired pending application.
    • Click Reject:
      • Actor clicks the "Reject" button.
      • The system displays a dialog prompting the Actor to enter a reason for rejection.
    • Submit Rejection:
      • Actor enters a reason in the rejection text field.
      • Actor clicks the "Reject" button on the dialog.
      • The system requests to process the application rejection from the server.
      • The server validates that the active user possesses a Staff or Admin role.
      • The server updates the instructor's application status to rejected in the database, including the reason.
      • The system displays a success notification message.
      • The system removes the rejected application row from the pending list dashboard.
  • Alternative Flow (Cancel Rejection)
    • If the Actor cancels the rejection:
      • Actor clicks the "Cancel" button on the rejection dialog.
      • The system closes the rejection dialog.
  • Alternative Flow (Submit Empty Rejection Reason)
    • If the Actor submits an empty reason:
      • Actor leaves the rejection reason blank and clicks the "Reject" button on the dialog.
      • The system displays a validation error message requiring a reason for rejection.
  • Postconditions
    • The instructor's status is updated to rejected in the database, along with the specified reason.
    • The rejected application is removed from the active pending list dashboard.
  • Error Handling
    • Session Invalid (Not Logged In): The system displays an authentication error message and redirects the Actor to the login page.
    • Unauthorized Access: The system detects that the Actor is not a Staff or Admin, displays an access denied error page, and terminates the flow.
    • Server Processing Error: The server fails to update the application status in the database, displays an error message regarding server processing failure, and returns to displaying the pending applications list page.
  • Security Measures
    • Role-based Access Control: Access to reject applications is restricted strictly to authenticated users with Staff or Admin roles.
    • Input Validation: The system requires a non-empty text value for the rejection reason before initiating the server request.
