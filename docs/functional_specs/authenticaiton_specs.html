a. Verify Email

Function Trigger: User enters the OTP code received in email and clicks the "Verify" / "Submit" button on the OTP verification form.

Function Description: Allows a User to verify their email address by submitting the One-Time Password (OTP) sent to their registered email. Upon validation, the system marks the account as verified and updates user session state.

Screen Layout:
[Placeholder for UI mockups]

Function Details

  • Preconditions
    • An OTP has been generated and sent to the User's email.
    • The User's email is stored in TempData.
  • Process
    • Verify Email Successfully:
      • User enters the OTP code received in email on the Verify OTP form.
      • User clicks the Verify button.
      • The system validates the OTP input form format.
      • The system retrieves the user's email from TempData.
      • The system submits a verification request with the email and OTP.
      • The system consumes and validates the OTP code.
      • The system retrieves the account by email and updates IsVerified = true in the database.
      • The system removes the verification flags from TempData.
      • The system updates the frontend IsVerified cookie to "true".
      • The system displays a verification success message and redirects the User to the Profile page.
  • Postconditions
    • The account's verification state is updated to true (IsVerified = true) in the database.
    • The system sets the IsVerified cookie to "true" and redirects the User to their Profile page with a success message.
  • Error Handling
    • Email Session Expired: The system detects that the email in TempData is empty or null, and redirects the User to the Login page.
    • Invalid OTP Format: The system detects that the entered OTP format is invalid, displays validation error messages, and returns to step 1.
    • Invalid Request State: The system detects that the verification payload is invalid or malformed, displays a request error message, and returns to step 1.
    • Incorrect or Expired OTP: The system detects that the OTP is incorrect or has expired, keeps the email and verification flags in TempData, displays the error message "Incorrect or expired OTP", and returns to step 1.
    • Account Not Found: The system detects that the account associated with the email does not exist in the database, displays an error message, and redirects the User to the Login page.
  • Security Measures
    • OTP Life Cycle: The OTP code must be consumed upon validation (one-time use only) and expires within 2 minutes of generation.
    • Session Cookie Sync: The system must update the IsVerified cookie to "true" immediately on successful verification to dynamically unlock shopping and checkout permissions without requiring a re-login.

b. Change Password

Function Trigger: User enters current password, new password, and confirms the new password, then clicks the "Change Password" button.

Function Description: Allows an authenticated User to update their password. The system verifies the current password, validates the new password, hashes it using BCrypt, saves it to the database, and forces a logout so the user must log in again with their new credentials.

Screen Layout:
[Placeholder for UI mockups]

Function Details

  • Preconditions
    • User must be logged in with a valid session (AccessToken).
  • Process
    • Change Password Successfully:
      • User accesses the Change Password page.
      • User enters current password, new password, and confirm new password.
      • User clicks the Change Password button.
      • The system validates the input formats.
      • The system verifies that the User has a valid AccessToken cookie.
      • The system validates that the new password matches the confirm new password.
      • The system retrieves the user's account details and checks that it is not an OAuth-linked account (has a non-empty password hash).
      • The system verifies that the current password matches the hashed password in the database.
      • The system hashes the new password using BCrypt and updates the account's password hash field.
      • The system saves the updated account details to the database.
      • The system calls the logout endpoint to revoke tokens.
      • The system deletes all local authentication cookies.
      • The system redirects the User to the Login page with a success message.
  • Postconditions
    • The user's account password hash is successfully updated in the database.
    • The user is logged out (all authentication cookies deleted) and redirected to the Login page with a success message.
  • Error Handling
    • Passwords Do Not Match: The system detects that the new password and confirm new password fields do not match, displays the error message "Password confirmation does not match", and returns to step 2.
    • OAuth Account Modification Denied: The system detects that the account lacks a password hash (registered via Google OAuth), displays an error message indicating that password change is not supported for external accounts, and returns to step 1.
    • Incorrect Current Password: The system detects that the entered current password does not match the stored hashed password, displays the error message "Current password is incorrect", and returns to step 2.
    • Database Update Failure: The system detects that saving the updated password hash failed (rows affected <= 0), displays an error message, and returns to step 2.
  • Security Measures
    • External Account Prevention: Users who registered using third-party auth (Google OAuth) are prohibited from changing password as they do not have a password hash stored locally.
    • Forced Logout: On a successful password change, the system must immediately revoke the refresh token, delete client-side cookies, and force the User to authenticate again to prevent active hijacked sessions.

c. Forget Password

Function Trigger: Actor accesses the Forgot Password page, enters their email, and clicks the "Submit" button.

Function Description: Allows an unauthenticated actor (User or Instructor) who has forgotten their password to perform a secure password reset. The system verifies the email, generates a 6-digit OTP code, sends it to the email, validates the entered OTP, and saves the new BCrypt-hashed password to the database.

Screen Layout:
[Placeholder for UI mockups]

Function Details

  • Preconditions
    • Actor must not be currently authenticated in the system.
  • Process
    • Reset Password Successfully:
      • Actor accesses the Forgot Password page.
      • Actor enters their registered Email address and clicks the Submit button.
      • The system queries the database to find the account by email.
      • The system validates that the account email exists, is verified, and is not linked to Google OAuth.
      • The system generates a 6-digit OTP code, stores it in the database with a 2-minute expiration, and sends it via email.
      • The system redirects the Actor to the Verify OTP page.
      • Actor enters the OTP code received in email and clicks the Verify button.
      • The system validates the OTP state in the database.
      • The system redirects the Actor to the Reset Password page.
      • Actor enters a new password and clicks the Reset button.
      • The system consumes (validates and deletes) the OTP code record.
      • The system hashes the new password using BCrypt and updates the password hash in the database.
      • The system redirects the Actor to the Login page with a "Password changed successfully" message.
  • Postconditions
    • A 6-digit OTP code is generated and emailed to the actor.
    • The account's password hash is successfully updated in the database and the OTP is consumed.
    • The system redirects the actor to the Login page with a success message.
  • Error Handling
    • Email Not Found: The system detects that the email does not exist in the database, displays the error message "Email not found", and returns to step 2.
    • Google Account Password Reset Denied: The system detects that the account is linked to Google OAuth, displays the error message "This account uses Google login", and returns to step 2.
    • Email Not Verified: The system detects that the account email has not been verified, displays the error message "Email is not verified", and returns to step 2.
    • Incorrect or Expired OTP: The system detects that the entered OTP is incorrect or has expired, displays the error message "Incorrect or expired OTP", and returns to step 7.
  • Security Measures
    • Verified Email Requirement: Only accounts that have completed email verification are permitted to use the forget password reset feature.
    • Google OAuth Lockout: Resetting passwords locally is blocked for accounts registered via Google OAuth because they authenticate externally and do not store a local password hash.
    • OTP Life Cycle: The reset password OTP expires within 2 minutes of generation and must be consumed (invalidated) immediately upon successful password modification to prevent reuse.

d. Logout

Function Trigger: Actor clicks the "Logout" button.

Function Description: Allows an authenticated actor (User, Instructor, Staff, or Admin) to safely terminate their session. The system invalidates the refresh token in the database, removes all local browser session cookies, and redirects the actor to the homepage in an unauthenticated state.

Screen Layout:
[Placeholder for UI mockups]

Function Details

  • Preconditions
    • Actor must be currently logged in and authenticated.
  • Process
    • Logout Successfully:
      • Actor clicks the Logout button.
      • The system sends a log out request to the frontend controller.
      • The frontend controller sends a POST request to the backend log out API with the active AccessToken.
      • The backend controller extracts the account ID from the token claims.
      • The system revokes the active RefreshToken for the account in the database.
      • The system clears backend API cookies.
      • The system deletes all frontend cookies.
      • The system redirects the Actor to the Homepage.
      • The system displays the homepage in an unauthenticated state showing Login and Register buttons.
  • Postconditions
    • The actor's refresh token is revoked in the database.
    • All local browser authentication and session cookies are deleted.
    • The system redirects the actor to the homepage.
  • Security Measures
    • Token Invalidation: Upon logout, the backend must immediately nullify/revoke the stored RefreshToken in the database to prevent token reuse attacks.
    • Cookie Clearance: Client-side cookies containing session context must be completely cleared.

e. Login with Google

Function Trigger: Actor clicks the "Sign in with Google" button on the Login page.

Function Description: Allows a User or Instructor to log in using their Google credentials via OAuth 2.0. If the email is logging in for the first time, a new account is registered automatically with default user privileges, auto-verified, and no password hash. Suspended accounts are blocked.

Screen Layout:
[Placeholder for UI mockups]

Function Details

  • Preconditions
    • Actor is on the Login page and not currently authenticated.
  • Process
    • Login with Google Successfully:
      • Actor clicks the “Sign in with Google” button.
      • The system triggers Google Identity authentication and opens the OAuth popup.
      • The Google Service returns the verified IdToken credential.
      • The system validates the Google IdToken structure and signature.
      • The system extracts the user profile: Email, Name, and Picture URL.
      • The system queries the database to find the account by email.
      • [Existing User] The system retrieves the account role.
      • The system verifies that the account does not have an active lockout.
      • The system generates a JWT AccessToken and RefreshToken.
      • The system appends HttpOnly cookies (AccessToken, RefreshToken) and client-readable cookies (UserName, AvatarUrl, UserRole, UserId, IsVerified = true).
      • The system redirects the Actor based on their role.
  • Alternative Flow (First Time Login - New User)
    • The system detects that the email does not exist in the database.
      • The system generates a unique username from the email prefix (appending numbers if the username is already taken).
      • The system instantiates a new Account and User entity.
      • The system saves the Account and User entities to the database.
      • The system retrieves the account role (returns to normal flow).
  • Postconditions
    • Authentication tokens and session cookies are set on the actor's browser.
    • If new, a verified Account and User record are saved to the database.
    • The system redirects the actor to the home page depending on their role.
  • Error Handling
    • Token Validation Failed: The system detects that the IdToken is invalid, malformed, or has expired, displays an authentication error message, and returns to step 1.
    • Account Suspended: The system detects that the account is suspended (has an active lockout), throws an UnauthorizedAccessException, displays a suspension warning indicating the lockout expiry date/time, and returns to step 1.
  • Security Measures
    • Unique Username Generation: For new Google sign-ups, the username is extracted from the email prefix. If the username is already taken, the system appends a numeric counter sequentially (e.g. johndoe1, johndoe2) until it is unique.
    • Auto-Verification: Accounts registered via Google OAuth are flagged as verified (IsVerified = true) immediately upon creation because Google has verified the email ownership.
    • Null Password Hash: Google OAuth accounts are initialized with no password hash (PasswordHash = null), bypassing local password authentication.
    • Suspension Block: If an active lockout exists for the account, the system denies login and prints the suspension lockout end date in the error message.
